### 1. Expand/Collapse du flyout amélioré (commit `a8a66be8`)
**Problème initial :** Cliquer sur l'en-tête d'un device pour déplier/replier paraissait « sec » (pas d'animation fluide).

**Solution implémentée :**
- **Rotation du chevron** : le chevron (glyphe `e010`) pivote à 180° de façon fluide au lieu de swapper entre deux glyphes. Animation `DoubleAnimation` sur `(UIElement.RenderTransform).(RotateTransform.Angle)` dans le `Style` (220ms, `CubicEase`).
  - **Piège résolu :** `Storyboard.TargetName` ne marche pas dans un `Style.Trigger` → utiliser property path au lieu d'un nom.
- **Re-open allégé** : le dépliage ferme/rouvre le flyout (architecture existante de la state-machine). Avant, ça rejouait le « pop » scale de toute la fenêtre → désagréable. Maintenant, on détecte la réouverture via `IsReopeningFromExpand` et on skip le pop ; seule la **cascade des lignes** porte la transition.
  - Nouvelle méthode `WindowAnimationLibrary.BeginFlyoutInstantShow` pour un show sans pop.
  - Fichiers touchés : `FlyoutWindow.xaml`, `FlyoutWindow.xaml.cs`, `FlyoutViewModel.cs`, `WindowAnimationLibrary.cs`.

**Verdict utilisateur :** ✅ « ok top ! J'aime beaucoup »

**Architecture découverte (important pour futur) :** Le dépliage ne déroule PAS la liste in-place ; il **ferme le flyout, fait `DoExpandCollapse` pendant qu'il est caché, puis rouvre**. Un accordéon de hauteur réel est donc quasi-impossible sans refonte. La solution pragmatique = améliorer la transition close→reopen.

### 3. Investigation MediaPopup (« le popup media n'a pas l'air de fonctionner »)
**Symptôme :** L'utilisateur survole l'icône tray, pas de popup média.

**Investigation :**
- Config registre : `MediaPopupEnabled = true`, `ShowOnlyWhenPlaying = true` → OK.
- Logs au premier démarrage : `"Startup: MediaPopup skipped (feature disabled)"` → `Settings.MediaPopupEnabled` retournait **false** au moment de `CompleteStartup`, alors que le registre contient **true**.

- **Workaround constaté :** Après **restart** de l'app, le log montre `"MediaPopup initialized at 815ms"` → fonctionne.

**Résolution empirique :** L'utilisateur confirme que le popup fonctionne après le restart (« ok ça fonctionne »).

## 🧠 Leçons / Pièges pour futures sessions

### 1. Animations WPF dans un `Style.Triggers`
**Piège :** Un `Storyboard` dans un `DataTrigger` de `Style` ne peut pas utiliser `Storyboard.TargetName="xxx"` (les noms ne sont pas accessibles depuis le scope du style).

**Solution :** Utiliser un property path relatif : `Storyboard.TargetProperty="(UIElement.RenderTransform).(RotateTransform.Angle)"` qui cible le `RenderTransform` de l'élément stylé lui-même.

**Exemple vécu :** Rotation du chevron (`FlyoutWindow.xaml` ligne ~224).

### 2. Architecture du dépliage (expand/collapse)
**Découverte :** Le dépliage ne modifie PAS la liste in-place. Flow réel :
1. Clic sur le bouton → `ExpandCollapse` command → `IsExpandingOrCollapsing = true` → `BeginClose`.
2. Le flyout se **ferme entièrement** (state → `Hidden`).
3. Dans `Hidden`, si `IsExpandingOrCollapsing`, appelle `DoExpandCollapse()` (qui insert/remove des devices dans `Devices`), puis `BeginOpen`.
4. Le flyout se **rouvre** avec la nouvelle liste.

**Implication :** Un accordéon de hauteur in-place (liste qui grandit) est impossible sans refonte de la state-machine. La solution pragmatique = améliorer la transition close→reopen (ce qu'on a fait).

### 3. Mesure de performance sur une `Window` WPF avec `WindowStyle=None` + cloak
**Piège :** Le flyout (`FlyoutWindow`) n'a pas de titre, utilise `WindowStyle=None`, et est cloak/uncloak dynamiquement → invisible pour `Get-Process` hwnd enum classique.

**Solution :** Mesure **idle** facile (juste lire `TotalProcessorTime` sur le process). Pour **flyout ouvert**, deux options :
- Mesure manuelle via Gestionnaire des tâches (utilisateur ouvre flyout, surveille CPU/GPU).
- Analyse du code source pour identifier les drivers de conso (ce qu'on a fait → cf. audit).

### 4. Incohérence des settings au démarrage (race condition suspectée)
**Bug latent :** `Settings.MediaPopupEnabled` peut retourner `false` au moment de `CompleteStartup` (App.xaml.cs ~359) alors que le registre contient `true` → le MediaPopup est skipped. Après restart, le read réussit → fonctionne.

**Hypothèse :** Le `RegistrySettingsBag` lit `HKCU:\Software\EarTrumpet` via `Registry.CurrentUser.CreateSubKey`. Si cette lecture échoue/est trop tôt, le `Get<bool>` retourne le `defaultValue` (qui est `true` pour `MediaPopupEnabled`... donc ça devrait marcher). **Contradiction non résolue** : le default est `true`, donc même un échec de lecture devrait retourner `true`. Pourtant le log dit "disabled". À creuser si le bug réapparaît.

**Workaround actuel :** Restart de l'app → fonctionne.
